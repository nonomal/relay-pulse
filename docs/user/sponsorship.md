## 2. 入驻与收录

### 2.1 申请流程

1. 使用 [API 自助测试](/selftest) 验证端点可用性（**必选**，未通过不予受理）
2. 通过 [GitHub Issue](https://github.com/prehisle/relay-pulse/issues/new?template=1-provider-suggestion.yml) 提交通道信息与赞助等级申请
3. 平台审核配置，如有问题在 Issue 中联调
4. 确认费用，通过邮件发送 API Key 并完成付款
5. 配置验证通过后上线监测，权益即时生效

### 2.2 通道命名规范

所有通道的 `channel` 字段须遵循统一命名格式：`<渠道类型>-<来源简写>`

| 标识 | 含义 | 示例 |
|:---:|------|------|
| **O** | 官方渠道 (Official) | `O-API`、`O-Web`、`O-AWS`、`O-GCP` |
| **R** | 逆向渠道 (Reverse) | `R-自定义名` |
| **M** | 混合渠道 (Mixed，默认) | `M-自定义名` |

**官方来源简写**：Web（网页订阅）、API（官方直连）、AWS（Bedrock）、GCP（Vertex AI）、App（合作应用）

> **过渡期**：现有未规范的通道名将在公告后 30 天内由平台统一迁移。过渡期内新旧名称并存，到期后以新名称为准。

### 2.3 付费方式

预付费制，按通道按月计费。具体支付方式以平台沟通确认为准。

### 2.4 API Key 授权

入驻需同意平台将所提供的 API Key 基于监测目的使用，包括但不限于：
- **可用性监测**：定期发起 API 请求，检测连通性与响应延迟
- **深度质量检测**：通过各种技术手段检测通道的服务质量

API Key 仅用于监测与展示目的，不用于其他用途。不同意授权则无法入驻。

## 3. 赞助升级与续费

### 3.1 升级条件

| 目标等级 | 最低收录天数 | 额外条件 |
|---------|:---:|------|
| public / signal / pulse | 无要求 | 入驻即可选择 |
| beacon（信标链路） | 7 天 | 位于主板 |
| backbone（骨干链路） | 15 天 | 位于主板 |
| core（核心链路） | 30 天 | 位于主板 |

- 收录天数以 `listed_since` 字段为准
- 「位于主板」要求通道当前未被自动移板至副板
- 升级申请通过 [赞助申请](https://github.com/prehisle/relay-pulse/issues/new?template=5-sponsorship.yml) 或邮件提交

### 3.2 到期与续费

通道配置 `expires_at` 字段（格式 `YYYY-MM-DD`）标记赞助到期日。到期日当天仍有效，次日起自动降级为 `pulse`（脉冲链路）并移入副板（secondary board）。到期前续费并更新 `expires_at` 可保持原等级与权益不变。

## 4. 质量治理

### 4.1 板块自动调整

系统每 30 分钟评估通道的 **7 天滚动可用率**：

- **主板 → 副板**：可用率低于 **50%** 时自动移入副板
- **副板 → 主板**：可用率恢复至 **55%** 以上时自动回到主板
- 评估需最少 **10 次**有效探测数据，数据不足时不做判断
- 自动移板仅影响板块归属，**不影响赞助等级和其他权益**

> 阈值间 5% 的缓冲带用于防止频繁切换。

### 4.2 置顶规则

适用于 `beacon` 及以上等级，需同时满足：无风险徽标、可用率 ≥ 90%（阈值可配置）。

**排序规则：**
1. 赞助级别降序：core > backbone > beacon
2. 同级别按可用率降序
3. 同可用率按响应延迟升序（低延迟优先）

**置顶限制：**
- 全局置顶数量受 `sponsor_pin.max_pinned` 限制（默认 10）
- 超出限额时按排序规则截断，未入选者按普通排序展示

**置顶时机：**
- 仅页面初始加载时生效，用户手动排序后以用户选择为准

## 5. 风险声明与条款

### 5.1 治理与反馈

- **反馈渠道**：[GitHub Issues](https://github.com/prehisle/relay-pulse/issues)
- **处理流程**：反馈提交 → 平台受理 → 调查核实 → 结果通知 → 申诉机制。平台将尽快处理，不承诺具体时限。
- **负向徽标**：仅针对严重问题（如欺诈、失联等），展示至问题解决，之后可申请撤销。适用于 `public` 及商业通道（`pulse` 及以上），`signal` 不适用。
- **义务**：各等级需接受用户反馈并配合问题调查，平台负责中立协调。

### 5.2 平台承诺

1. **数据真实性**：保证不篡改监测数据
2. **中立协调**：反馈处理中保持中立，提供沟通支持
3. **规则透明**：规则变更提前公示通知
4. **隐私保护**：不泄露赞助方敏感信息

### 5.3 风险提示与免责

1. **数据滞后性**：监测数据反映既往技术状态，不构成对未来可用性的保证。
2. **服务定位**：RelayPulse 仅提供技术监测服务，不提供交易撮合、资金托管或担保服务。
3. **赞助说明**：赞助标识仅代表对应等级的展示与监测权益，不构成对服务商信誉或履约能力的背书。
4. **责任边界**：用户与第三方服务商之间的交易风险由双方自行承担。平台不对第三方服务的质量、履约或资金安全承担担保责任。
5. **资金安全提示**：中转 API 行业存在经营风险，建议遵循「用多少充多少」原则，避免大额预充值。高可用率仅代表技术稳定性，不代表商家信誉。

### 5.4 规则变更与协议确认

- 规则变更将通过网站公告通知，既有权益保持至赞助到期。
- 本规则最终解释权归 RelayPulse 所有。
- **完成赞助即视为已阅读并同意本规则全部条款。**
