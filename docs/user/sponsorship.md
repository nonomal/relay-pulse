## 3. 置顶规则

适用于 `beacon` 及以上等级，需同时满足：无风险徽标、可用率 ≥ 90%（阈值可配置）。

**排序规则：**
1. 赞助级别降序：core > backbone > beacon
2. 同级别按可用率降序
3. 同可用率按响应延迟升序（低延迟优先）

**置顶限制：**
- 全局置顶数量受 `sponsor_pin.max_pinned` 限制（默认 10）
- 超出限额时按排序规则截断，未入选者按普通排序展示

**置顶时机：**
- 仅页面初始加载时生效，用户手动排序后以用户选择为准

## 4. 入驻与续费

### 申请流程

1. 通过 [GitHub Issue](https://github.com/prehisle/relay-pulse/issues/new?template=1-provider-suggestion.yml) 提交通道信息与赞助等级申请
2. 平台审核配置，如有问题在 Issue 中联调
3. 确认费用，通过邮件发送 API Key 并完成付款
4. 配置验证通过后上线监测，权益即时生效

### 付费方式

预付费制，按通道按月计费。具体支付方式以平台沟通确认为准。

### 到期与续费

通道配置 `expires_at` 字段（格式 `YYYY-MM-DD`）标记赞助到期日。到期日当天仍有效，次日起自动降级为 `pulse`（脉冲链路）并移入副板（secondary board）。到期前续费并更新 `expires_at` 可保持原等级与权益不变。

此外，可用率低于 50% 的通道也将自动移入副板（仅影响板块归属，不影响赞助等级）。

### API Key 授权

入驻需同意平台将所提供的 API Key 用于以下目的：
- **可用性监测**：定期发起 API 请求，检测连通性与响应延迟
- **质量监测**：通过各种技术手段检测通道的服务质量

API Key 仅用于监测与展示目的，不用于其他用途。不同意授权则无法入驻。

## 5. 风险声明与条款

### 5.1 治理与反馈

- **反馈渠道**：[GitHub Issues](https://github.com/prehisle/relay-pulse/issues)
- **处理流程**：反馈提交 → 平台受理 → 调查核实 → 结果通知 → 申诉机制。平台将尽快处理，不承诺具体时限。
- **负向徽标**：仅针对严重问题（如欺诈、失联等），展示至问题解决，之后可申请撤销。适用于 `public` 及商业通道（`pulse` 及以上），`signal` 不适用。
- **义务**：各等级需接受用户反馈并配合问题调查，平台负责中立协调。

### 5.2 平台承诺

1. **数据真实性**：保证不篡改监测数据
2. **中立协调**：反馈处理中保持中立，提供沟通支持
3. **规则透明**：规则变更提前公示通知
4. **隐私保护**：不泄露赞助方敏感信息

### 5.3 风险提示与免责

1. **数据滞后性**：监测数据反映既往技术状态，不构成对未来可用性的保证。
2. **服务定位**：RelayPulse 仅提供技术监测服务，不提供交易撮合、资金托管或担保服务。
3. **赞助说明**：赞助标识仅代表对应等级的展示与监测权益，不构成对服务商信誉或履约能力的背书。
4. **责任边界**：用户与第三方服务商之间的交易风险由双方自行承担。平台仅在自身故意或重大过失造成损害时依法承担责任。
5. **资金安全提示**：中转 API 行业存在经营风险，建议遵循「用多少充多少」原则，避免大额预充值。高可用率仅代表技术稳定性，不代表商家信誉。

### 5.4 规则变更与协议确认

- 规则变更将通过网站公告通知，既有权益保持至赞助到期。
